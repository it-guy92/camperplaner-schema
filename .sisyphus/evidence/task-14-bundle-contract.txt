# Purpose: Define new get_place_source_bundle RPC contract aligned with property tables

## CONTRACT SUMMARY

RPC: get_place_source_bundle(place_id BIGINT) RETURNS JSONB

The RPC is redesigned to return minimal `places` data plus the new aligned property
tables instead of the legacy source-family parent tables.

---

## OLD CONTRACT (DEPRECATED)

Returns JSONB with keys:
- `base`: Full places row (SELECT to_jsonb(p.*) â€” includes ALL 30+ columns)
- `osm`: Current osm_source row (parent table, raw OSM data)
- `llm`: Current place_llm_enrichments row (parent table, LLM processing metadata)
- `google`: Current place_google_sources row (parent table, Google API cache)
- `user_aggregates`: { review_count, avg_rating, favorite_count }

### Problems with Old Contract:
1. `base` includes 20+ deprecated business columns (has_toilet, website, phone, etc.)
2. `osm` returns raw OSM source tracking data, not aligned properties
3. `llm` returns LLM processing metadata (tokens, cost, status), not extracted facts
4. `google` returns Google API cache metadata, not aligned properties
5. No user property corrections
6. No Google reviews or photos (1:n child data excluded)

---

## NEW CONTRACT (TARGET)

Returns JSONB with keys:
- `base`: Minimal places row (id, lat, lon, is_active, created_at, updated_at)
- `osm_properties`: Current place_osm_properties row (aligned 55+5 columns)
- `google_properties`: Current place_google_properties row (aligned 55+6 columns)
- `llm_properties`: Current place_llm_properties row (aligned 55+6 columns)
- `user_properties`: Current place_user_properties row (aligned 55+1 columns, per-user)
- `google_reviews`: Array of place_google_reviews for current google_source_id
- `google_photos`: Array of place_google_photos for current google_source_id
- `user_aggregates`: { review_count, avg_rating, favorite_count }

### Key Changes:
1. `base` reduced from 30+ columns to 6 columns (minimal places)
2. `osm` â†’ `osm_properties`: Aligned property table with typed boolean columns
3. `llm` â†’ `llm_properties`: Extracted facts as columns, not processing metadata
4. `google` â†’ `google_properties`: Aligned property table with typed boolean columns
5. NEW `user_properties`: User corrections with highest resolution priority
6. NEW `google_reviews`: 1:n child data included as array
7. NEW `google_photos`: 1:n child data included as array
8. `user_aggregates`: Unchanged (review_count, avg_rating, favorite_count)

## RESOLUTION PRIORITY

When resolving a property value from the bundle:
1. user_properties (if non-NULL) â€” human corrections, highest priority
2. osm_properties (if non-NULL) â€” community-verified data
3. google_properties (if non-NULL) â€” commercial source
4. llm_properties (if non-NULL) â€” AI-generated, lowest priority

This priority is NOT enforced by the RPC â€” it returns all sources.
The consumer (product/worker) implements the resolution logic.

## LEGACY TABLES IN BUNDLE

### REPLACED (no longer returned as source keys):
- `osm_source` â†’ replaced by `place_osm_properties`
  - Rationale: osm_source is raw import tracking data; properties are aligned business data
- `place_llm_enrichments` â†’ replaced by `place_llm_properties`
  - Rationale: enrichments are processing metadata; properties are extracted facts
- `place_google_sources` â†’ replaced by `place_google_properties`
  - Rationale: sources are API cache metadata; properties are aligned business data

### RETAINED (included as separate keys):
- `place_google_reviews` â†’ `google_reviews` key (1:n, cannot flatten)
- `place_google_photos` â†’ `google_photos` key (1:n, cannot flatten)

### RETAINED (unchanged):
- `campsite_reviews` â†’ used for user_aggregates.review_count, user_aggregates.avg_rating
- `favorites` â†’ used for user_aggregates.favorite_count
