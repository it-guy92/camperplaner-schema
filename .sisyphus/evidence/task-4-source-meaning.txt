SECTION 1: GOOGLE AMENITY MEANING PRESERVATION
================================================================================

The place_google_amenities table uses a normalized fact model with:
- amenity_key: Standardized identifier (e.g., 'electricity', 'water', 'wifi')
- value_boolean/value_text/value_numeric: Typed values
- is_verified: Whether explicitly confirmed vs inferred
- confidence_score: Confidence level (1.0 = explicit, lower = inferred)
- google_feature_type: Original Google type that indicated this amenity
- source_section: Which part of Google response provided the info

FLATTENING STRATEGY:
Each amenity_key maps to a dedicated boolean column in place_google_properties.
This preserves the core meaning (amenity present/absent) while simplifying the schema.

MEANING PRESERVED:
✓ Amenity presence/absence (boolean value)
✓ Amenity identity (amenity_key → column name mapping is deterministic)

MEANING DISCARDED:
✗ Verification status (is_verified) - not needed in simplified model
✗ Per-amenity confidence (confidence_score) - aggregated into overall data_confidence
✗ Source provenance (google_feature_type, source_section) - process metadata

RATIONALE:
The simplified model prioritizes query performance and schema clarity over
provenance detail. The amenity boolean is the business-meaningful fact.
Verification status and source section are operational metadata useful for
debugging but not for application logic.

SECTION 4: LLM SOURCE MEANING PRESERVATION
================================================================================

The place_llm_sources table stores URLs cited by the LLM with:
- source_url: Full URL of the source
- source_domain: Domain extracted from URL
- source_kind: Type of source (website, google_reviews, osm, user_submission, other)
- trusted: Whether domain is in trusted list
- relevance_score: Per-URL relevance (0-1)

FLATTENING STRATEGY:
All source_url values are aggregated into source_urls text[] in place_llm_properties.
Trust information contributes to the overall trust_score.

MEANING PRESERVED:
✓ Complete URL list (source_urls array)
✓ Trust signal (contributes to trust_score)

MEANING DISCARDED:
✗ Source domain (source_domain) - derivable from URL
✗ Source kind (source_kind) - classification detail
✗ Per-URL relevance (relevance_score) - aggregated into trust_score

RATIONALE:
The URL itself contains the domain. Source kind is a classification detail
not needed for application logic. Per-URL relevance is useful for ranking
but the simplified model uses a single trust score.

SECTION 5: CROSS-SOURCE MEANING PRESERVATION
================================================================================

The new property tables preserve cross-source meaning through:

1. ALIGNED SHARED COLUMNS
   All four property tables (OSM, Google, LLM, User) share the same set of
   property columns (has_toilet, has_shower, website, phone, etc.).
   This enables deterministic resolution priority: User → OSM → Google → LLM.

2. SOURCE-SPECIFIC METADATA
   Each property table includes source-specific columns:
   - OSM: osm_type, osm_id, tags
   - Google: google_place_id, rating, review_count, business_status
   - LLM: provider, model, summary_de, hallucination_risk
   - User: user_id, correction_reason

3. CURRENT-ROW SEMANTICS
   Each source maintains one current row per place (or per user for User properties).
   This preserves the temporal aspect of source data without cluttering the schema.

4. NULL = NO CLAIM SEMANTICS
   Booleans default to NULL (not false) to distinguish "source claims false"
   from "source made no claim". This preserves the semantic difference between
   explicit negative claims and absence of information.
