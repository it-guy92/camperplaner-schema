1. CURRENT-ROW CONSTRAINT SPECIFICATIONS
================================================================================

The following partial unique indexes enforce "one current row per X" semantics
using PostgreSQL partial unique indexes. This pattern ensures:
- Multiple historical rows can coexist (is_current = false)
- Only one row per scope can have is_current = true
- The constraint is automatically enforced by PostgreSQL

--------------------------------------------------------------------------------
1.1 place_osm_properties
--------------------------------------------------------------------------------

Constraint Name: uidx_osm_properties_place_current

SQL:
  CREATE UNIQUE INDEX uidx_osm_properties_place_current 
      ON place_osm_properties(place_id) 
      WHERE is_current = true;

Purpose: Ensures only one current OSM property row per place
Scope: One row per place_id when is_current = true

SQL:
  CREATE UNIQUE INDEX uidx_user_properties_place_user_current 
      ON place_user_properties(place_id, user_id) 
      WHERE is_current = true;

Purpose: Ensures only one current user property row per (place, user) pair
Scope: One row per (place_id, user_id) when is_current = true

Note: Different from source tables because user corrections are per-user,
not global. Each user can have one "current" correction per place.

3. CONSTRAINT DESIGN PATTERN
================================================================================

The partial unique index pattern is chosen over:
- Trigger-based solutions (more complex, potential race conditions)
- Application-level enforcement (not enforceable at DB level)
- Full unique constraints (would prevent historical rows)

Why Partial Unique Indexes:
1. PostgreSQL natively enforces this pattern
2. Automatically handles concurrent inserts
3. No application logic required
4. Clear semantics: WHERE clause defines the partial uniqueness

Pattern Source:
- osm_source: uidx_osm_source_place_current
- place_google_sources: uidx_google_sources_place_current  
- place_llm_enrichments: uidx_llm_enrichments_place_current_completed

================================================================================
5. ROLLBACK CONSIDERATIONS
================================================================================

To rollback current-row constraints:
- Simply DROP the partial unique indexes
- No data migration required
- Historical rows remain intact

Example rollback:
  DROP INDEX IF EXISTS uidx_osm_properties_place_current;
  DROP INDEX IF EXISTS uidx_google_properties_place_current;
  DROP INDEX IF EXISTS uidx_llm_properties_place_current;
  DROP INDEX IF EXISTS uidx_user_properties_place_user_current;
