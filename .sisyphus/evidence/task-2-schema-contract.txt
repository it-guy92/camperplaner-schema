SECTION 3: BOOLEAN SEMANTICS
================================================================================

CRITICAL RULE: NULL vs FALSE

For ALL boolean property columns across ALL four tables:

  NULL  = The source made NO CLAIM about this property.
          The source did not mention it, did not assert it, did not deny it.
          This is the DEFAULT state for all boolean columns.

  FALSE = The source EXPLICITLY DENIED this property.
          The source stated this feature is NOT present or NOT allowed.
          This requires affirmative evidence from the source.

  TRUE  = The source EXPLICITLY CONFIRMED this property.
          The source stated this feature IS present or IS allowed.

IMPLICATIONS:
- Boolean columns MUST be nullable (no NOT NULL constraint)
- Boolean columns MUST NOT have DEFAULT false
- Backfill logic MUST preserve NULL when source has no claim
- Read resolution MUST distinguish "unknown" (NULL) from "no" (false)

EXAMPLES:
  OSM tags say "toilets=yes"    → has_restrooms = TRUE
  OSM tags say "toilets=no"     → has_restrooms = FALSE
  OSM tags do not mention toilets → has_restrooms = NULL

  Google amenities list "Restroom" → has_restrooms = TRUE
  Google amenities do not list it  → has_restrooms = NULL (NOT false)

  LLM summary says "no WiFi"    → has_wifi = FALSE
  LLM summary says "free WiFi"  → has_wifi = TRUE
  LLM summary doesn't mention   → has_wifi = NULL

================================================================================
SECTION 4: CURRENT-ROW SEMANTICS
================================================================================

OSM/Google/LLM Properties:
- ONE current row per place_id
- Enforced via partial unique index: WHERE is_current = true
- When new data arrives, set old row is_current=false, insert new row is_current=true

User Properties:
- ONE current row per (place_id, user_id)
- Enforced via partial unique index: WHERE is_current = true
- Each user can submit multiple corrections over time; only latest is current

================================================================================
SECTION 5: READ RESOLUTION PRIORITY
================================================================================

When multiple sources provide data for the same field, resolve in this order:

  1. User properties (highest priority - human correction)
  2. OSM properties (community-verified data)
  3. Google properties (commercial API data)
  4. LLM properties (lowest priority - AI-generated)

This priority applies to ALL shared columns unless a consumer explicitly
overrides for a specific use case.
