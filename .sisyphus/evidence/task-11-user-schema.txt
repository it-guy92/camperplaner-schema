SECTION 1: USER SCHEMA OVERVIEW
================================================================================

The place_user_properties table stores user-submitted property corrections for
places. Each user can submit multiple corrections over time, with only the
latest correction being "current".

This table follows the SHARED PROPERTY SCHEMA CONTRACT (task-2-schema-contract.txt)
plus one user-specific column.

SECTION 3: FOREIGN KEY RELATIONSHIPS
================================================================================

place_user_properties has two foreign keys:

1. place_id → places(id)
   - ON DELETE CASCADE: If a place is deleted, all user corrections are deleted
   - Ensures referential integrity with the canonical places table

2. user_id → profiles(id)
   - ON DELETE CASCADE: If a user profile is deleted, all their corrections are deleted
   - Ensures referential integrity with the profiles table

NOTE: The current migration (20260318200000_add_property_tables.sql) is MISSING
the FK constraint for user_id. This must be added in a follow-up migration.

================================================================================
SECTION 4: COLUMN PURPOSES
================================================================================

user_id:
    - Identifies which user submitted the correction
    - Required (NOT NULL) - every correction must be attributed
    - Enables per-user correction history tracking
    - Enables user-specific read resolution (show user's own corrections)

source_updated_at:
    - When the user submitted or updated this correction
    - NULL until explicitly set by user action
    - Distinct from updated_at (which tracks any row modification)

is_current:
    - TRUE = This is the user's latest correction for this place
    - FALSE = This is an older correction, superseded by a newer one
    - Only ONE current row allowed per (place_id, user_id) combination

All other columns follow the shared schema contract (task-2-schema-contract.txt).

================================================================================
SECTION 5: DIFFERENCE FROM SHARED SCHEMA
================================================================================

The ONLY structural difference between place_user_properties and the other
property tables (place_osm_properties, place_google_properties, place_llm_properties)
is the addition of user_id.

All other columns - their types, nullability, defaults - are IDENTICAL to the
shared schema contract. This ensures:
- Consistent read resolution across all sources
- Uniform boolean semantics (NULL = unknown, TRUE = confirmed, FALSE = denied)
- Simple UNION queries across all property tables

================================================================================
SECTION 6: BOOLEAN SEMANTICS (APPLIES TO ALL SHARED COLUMNS)
================================================================================

NULL  = User did NOT claim anything about this property
       (user left field blank, didn't know, or source had no info)

FALSE = User EXPLICITLY DENIED this property
       (user stated feature is NOT present or NOT allowed)

TRUE  = User EXPLICITLY CONFIRMED this property
       (user stated feature IS present or IS allowed)
