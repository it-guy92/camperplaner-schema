================================================================================
SECTION 1: GATE FRAMEWORK
================================================================================

Each deprecated table has a cleanup gate with 4 mandatory checks:

1. BACKFILL VERIFIED: Target property table has data for all places that had data in deprecated table
2. READ CUTOVER VERIFIED: No application code reads from deprecated table
3. WRITE CUTOVER VERIFIED: No application code writes to deprecated table
4. ROLLBACK READINESS: Pre-drop backup exists and is verified

ALL 4 checks must pass before a table enters the drop queue.

================================================================================
SECTION 4: GATE FAILURE PROTOCOL
================================================================================

If ANY gate check fails:

1. HALT: Do not proceed with drop for that table
2. LOG: Record failure in cutover_audit_log with:
   - table_name
   - gate_check (e.g., "Gate 4.1 — BACKFILL VERIFIED")
   - failure_reason
   - expected_value
   - actual_value
3. FIX: Address the root cause
4. RE-EVALUATE: Re-run the failed gate check
5. RESUME: Only proceed when all gates for that wave pass

Gate failures are NOT warnings — they are hard stops.
