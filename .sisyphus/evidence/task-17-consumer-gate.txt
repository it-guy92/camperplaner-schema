# Task 17: Consumer Gate and Cutover Timing

## Overview
This file documents the explicit gates and timing requirements for type regeneration, downstream sync, and consumer code cutover. It ensures consumers stop reading deprecated fields only after migration is complete and types are synced.

---

## Type Regeneration Command

```bash
# From schema repository root
supabase gen types typescript --local > generated/database.types.ts
```

**Critical Timing:** Run AFTER migration executes, NOT before.

## Type Sync Timing

### Standard Timing

| Event | When | Who |
|-------|------|-----|
| Types generated | After migration executes | Schema Repo |
| Types committed | Immediately after generation | Schema Repo |
| Types synced | Within 1 business day | Product/Worker |

### Accelerated Timing (Urgent)

For hotfixes (per AGENTS.md Section "Hotfix Escalation Path"):
- Types committed within hours, not days
- Consumers sync same day
- Fast-track approval available

## Consumer Code Cutover Rules

### Phase 1: Start Writing to New Schema

**Before Cutover:**
- Old code still writes to legacy tables
- New code can read new tables

**Cutover Point:** Worker deploys with new write logic

**After Cutover:**
- New code writes to new tables
- Legacy writes can continue (dual-write period)

### Phase 2: Switch to New Views/RPCs

**Before Cutover:**
- Product reads from legacy views
- Worker calls legacy functions

**Cutover Point:** Both repos deploy with new view/function calls

**After Cutover:**
- All reads go through new views
- All function calls use new signatures

### Phase 3: Stop Reading Deprecated Fields

**Before Cutover:**
- Product code may still reference deprecated fields
- Worker code may still reference deprecated fields

**Cutover Point:** Gate G3.3 AND G3.4 satisfied (both confirmed removal)

**After Cutover:**
- Schema drops deprecated columns
- No consumer code references them

## Validation Checkpoints

### Product Validation

```bash
# After syncing types
cd apps/web
npm run typecheck
npm test

# Verify no deprecated field references
grep -r "deprecated_field_name" --include="*.ts" --include="*.tsx" src/
```

### Worker Validation

```bash
# After syncing types
npm run typecheck
npm test

# Verify no deprecated field references
grep -r "deprecated_field_name" --include="*.ts" src/
```

## Rollback Triggers

| Trigger | Condition | Action |
|---------|-----------|--------|
| Migration fails | Staging deployment fails | Revert, investigate |
| Type gen fails | supabase gen exits non-zero | Revert migration |
| Product compile error | Type errors after sync | Revert types, investigate |
| Worker compile error | Type errors after sync | Revert types, investigate |
| Product test failure | Tests fail after sync | Block Phase 3 until fixed |
| Worker test failure | Tests fail after sync | Block Phase 3 until fixed |

---

## Communication Requirements

### Before Each Phase

1. Schema repo posts issue: "Upcoming Phase N migration"
2. Schema repo tags: @product-maintainer, @worker-maintainer
3. Impact assessment included

### After Each Phase

1. Schema repo posts: "Phase N merged"
2. Includes migration details
3. Includes sync commands
4. Includes deadline (1 business day)

### Phase 3 Specific

1. Schema repo verifies evidence artifacts
2. Schema repo verifies observation window
3. Schema repo asks: "Are deprecated refs removed?"
4. Product confirms: "Yes, all refs removed"
5. Worker confirms: "Yes, all refs removed"
6. ONLY THEN: Schema repo proceeds with cleanup

---

## Cutover Decision Matrix

| Scenario | Can Proceed? | Condition |
|----------|--------------|-----------|
| Types synced, product updated, worker pending | Worker only | Product gate passed |
| Types synced, worker updated, product pending | Product only | Worker gate passed |
| Types synced, both updated, tests passing | YES | All gates passed |
| Phase 3, product done, worker pending | NO | Blocked by worker |
| Phase 3, observation not complete | NO | Blocked by time |
| Phase 3, evidence missing | NO | Blocked by artifacts |
